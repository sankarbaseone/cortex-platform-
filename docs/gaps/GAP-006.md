# GAP-006: T-003 file list omits nyxgrpc, which ECD-002/RFC-006 assume exists
Status: Accepted (deferred)
## Affected RFC/ECD
`05-manifests/build-order.yaml` (T-003 `libs-core` files: `nyxerr, nyxpg, nyxbus, nyxotel, nyxhttp, nyxrun, nyxclock, config` — no `nyxgrpc`) vs. `02-ecd/ECD-002-source-code-construction-plan.md` ("`internal/adapters/grpc/server.go` | generated svc impl → domain calls; interceptors from `nyxgrpc` (auth, tenant, deadline, otel, recovery, ratelimit — order fixed exactly that)") vs. `01-rfc/RFC-006-api-sdk-specification.md` F.5 ("Interceptors (all services): auth, tenant-context, deadline-propagation, otel-trace, panic-recovery, rate-limit").
## The ambiguity
Two frozen documents assume a `nyxgrpc` library exists and is where every service's gRPC interceptor chain lives, but the machine-readable `build-order.yaml` — authoritative per the established authority order (GAP-003) — does not list `nyxgrpc` anywhere in T-003's (or any other task's) file set.
## Reason
Likely an omission in build-order.yaml's T-003 entry rather than a deliberate decision to drop the library — no ADR or RFC note argues against a shared interceptor-chain package, and RFC-014 G.0's uniform service contract (fixed interceptor order, breakers, etc.) only makes sense as shared code.
## Resolution
Deferred, not re-escalated (same posture as GAP-002/GAP-005): T-003 built exactly the 8 libs build-order.yaml names. `nyxgrpc` was not created. Whichever task first wires a real gRPC server (T-101 kernel-registry is the first candidate per S1-ingest-spine) must either add `libs/go/nyxgrpc` at that point or fold the interceptor chain directly into that service if a second consumer never materializes — but per RFC-014 G.0 the interceptor order is meant to be uniform across all 16 services, so a shared library is almost certainly still correct.
## Narrowest-reading fallback
Applied: T-003 delivered `nyxerr, nyxpg, nyxbus, nyxotel, nyxhttp, nyxrun, nyxclock, config` only. `libs/go/nyxhttp.NewBreaker` exists (RFC-014 G.0 outbound breaker requirement, which *is* in scope since it applies to any outbound sync call, not gRPC-server-specific); `nyxgrpc`-specific interceptor chain code does not.
