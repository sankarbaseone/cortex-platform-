# GAP-005: `analyzers/` (ECD-001) vs `dataplane/` + top-level `crates/` (build-order.yaml, shipped README)
Status: Accepted (resolved per founder's stated authority order, not re-escalated)
## Affected RFC/ECD
`02-ecd/ECD-001-repository-construction-plan.md` §2 (tree shows `analyzers/{collector,edge-gateway,compiler-analyzer/crates/{...},runtime-analyzer,crates/common/{...}}`) vs. `05-manifests/build-order.yaml` (T-103: `dataplane/edge-gateway`, `dataplane/collector`; T-201/T-202/T-203: `crates/canonicalizer`, `crates/ptx-parser`, `crates/sass-decoder`, `crates/kes` at repo-root, not nested under `analyzers/`; T-204: `dataplane/compiler-analyzer`) vs. the shipped `README.md`'s own "Repo map (ECD-001)" line, which independently says "`crates/` Rust analyzers ... `dataplane/` collector + edge-gateway + analyzers" — agreeing with build-order.yaml, not with ECD-001's own diagram.
## The ambiguity
ECD-001's tree diagram nests all Rust code under `analyzers/`. build-order.yaml's task file lists and the shipped README (both distinct, independently-authored documents) instead use `dataplane/` for the analyzer services and a repo-root `crates/` for the Rust libraries. This is a two-against-one disagreement with ECD-001's own diagram, discovered while scoping T-001's directory skeleton.
## Reason
Apparent drift between the ECD-001 diagram (possibly an earlier draft) and the machine-readable build-order.yaml + shipped README (likely a later, corrected pass).
## Resolution
Per the founder's stated authority order (machine-readable manifests > RFCs > ECDs > ITP > README), and given build-order.yaml plus the shipped README both independently agree against ECD-001's diagram alone, `dataplane/` + top-level `crates/` is treated as correct going forward. Not re-escalated to the founder, consistent with the "documentation inconsistency, don't stop implementation" instruction already given for the analogous `registries/` omission.
## Narrowest-reading fallback
Applied: T-001 created neither `analyzers/` nor `dataplane/`/`crates/` — this naming is load-bearing for T-103/T-201-T-206, none of which are in T-001's scope. Whichever task creates that code first should follow `dataplane/` + `crates/`.
