# GAP-004: docs/ layout — specification archive vs. ECD-001 live tree
Status: Accepted (founder-resolved)
## Affected RFC/ECD
`01-rfc/RFC-012-claude-code-implementation-guide.md` §O.1 (canonical CLAUDE.md text: "Read RFC-000..014 in /docs/rfcs"); shipped `README.md` ("See `docs/rfcs/` ... `docs/ecds/`"); vs. the already-committed, tagged (`spec-v1.0`) archive at `docs/specification/v1.0/{00-blueprint,01-rfc,02-ecd,...}`.
## The ambiguity
The frozen corpus's own canonical CLAUDE.md/README text expects RFCs/ECDs to live at repo-root `docs/rfcs/` and `docs/ecds/`. This does not match the numbered-prefix archive structure (`docs/specification/v1.0/01-rfc/`, `02-ecd/`, etc.) built and approved in the prior specification-population phase and tagged `spec-v1.0`.
## Reason
The archive structure was a staging/traceability convention invented during specification population, not something named anywhere in the frozen RFC/ECD/ITP corpus itself (confirmed: the string "specification/v1.0" appears nowhere in the frozen documents).
## Resolution
Founder-approved: `docs/specification/v1.0/` is the immutable Specification Archive — for traceability, auditing, and architecture reference only. It is not part of the runtime repository structure and must not be modified during implementation. The live `docs/rfcs/`, `docs/ecds/`, `docs/runbooks/`, `docs/adr/` etc. defined by ECD-001 are a separate, distinct tree, populated independently by later tasks as needed — not by T-001, whose file list does not include them.
## Narrowest-reading fallback
Applied: T-001 created empty `docs/rfcs/`, `docs/ecds/`, `docs/runbooks/`, `docs/adr/` directories (skeleton only, per ECD-001 §2), with zero content copied from the archive. `docs/specification/v1.0/` was left untouched.
